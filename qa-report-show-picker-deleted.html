Corregido — ya desplegado en dev El picker de Show del detalle de Media ya no incluye shows con status DELETE.
A tener en cuenta: el PR sm2#8443 figura sin mergear en GitHub (branch fix/issue-8442 → staging) al momento de esta verificación. Sin embargo, el comportamiento observado en vivo en dev coincide exactamente con el fix del PR (la petición del picker ya no lleva all=true) — el código probablemente ya fue promovido a staging/dev por otra vía. Confirmar el estado real del PR antes de dar el fix por cerrado formalmente.

Reporte original (cliente)

"Tenemos un reporte del cliente, donde indica que, al ir a media en el apartado show, por ejemplo si se va a asignar el show ‘Capital, la Bolsa y la Vida’ se está listando tres veces, aunque en el módulo show solamente existe 1 con este nombre. Al revisar lo que sucede es que la petición que se trae aquí es platform.mediastre.am/api/show/list?all=true — esto genera este comportamiento, al parecer trae shows que ya no existen, esto no pasa solo con este show, pasa con más shows. Cuando se hace la petición platform.mediastre.am/api/show/list, esta sí trae un único resultado."

Causa raíz

El endpoint GET /api/show/list filtra por status: 'OK' salvo que se pida ?all=true, en cuyo caso devuelve shows de cualquier status — incluidos los marcados DELETE. El picker de Show del detalle de Media (show_options.coffee) pedía ?all=true específicamente cuando el selector es para un media. Si varios shows borrados comparten título entre sí o con uno activo (algo común: recrear un show con el mismo nombre, o datos residuales de pruebas), el dropdown los lista como entradas separadas y visualmente idénticas. 

module.exports = async function (req, res) {
  ...
  const query = { account }
  if (req.query.all !== 'true') {
    query.status = 'OK'
  }
  ...
}

src/server/routes/api/show/list.js (sm2) — el filtro real que all=true desactiva.

El fix (sm2#8443)

  1. Cliente (show_options.coffee): saca el ?all=true que se agregaba cuando el picker es para media. Coincide con lo que reportó el cliente: sin all=true el endpoint sí trae un único resultado por show.
  2. Servidor (media/update.js, guarda defensiva): con los shows borrados ya invisibles en el picker, un guardado sin selección podía desvincular en silencio un media de un show borrado. Si al guardar se limpia show_info.showId y el show previo tiene status: DELETE, el server restaura el link anterior en vez de dejarlo en null.
