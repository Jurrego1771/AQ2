Sesión de exploración + cobertura del flujo self-service de API tokens (/settings/api), historia US-036 (scope read/write). A diferencia del gate por TokenProfile (US-035, bloqueado por is_admin de plataforma), el acceso read/write sí es explorable con la cuenta QA — y desde hoy la cuenta puede crear, listar, actualizar y borrar sus tokens (permiso api_tokens + delete concedido 2026-07-14), lo que permite tests self-contained.

Veredicto: el scope funciona correctamente (read rechaza escrituras, write escribe, borrado/deshabilitado invalidan al instante). 1 bug real filado (AQ2#55, medium): un token read-only en POST /api/show devuelve 500 en vez de 403. Además 4 mejoras no bloqueantes documentadas. 14 tests nuevos (13 verdes + 1 rojo-esperado vivo del bug).

Qué se probó

  • Creación de tokens read y write vía API (sesión) y vía UI (/settings/api).
  • Scope de lectura: token read → GET show/media (200) y POST (403).
  • Scope de escritura: token write → GET + POST /api/show (200).
  • Ciclo de vida: borrar y deshabilitar (toggle) un token → deja de autenticar al instante.
  • Casos borde: crear sin descripción, access como objeto vs string.
  • UX/heurísticas: feedback de creación, confirmación de borrado, nombres accesibles de los botones de acción.

Qué NO se cubrió (huecos de cobertura) (2)

gap GAP-TOKENPROFILE — Gate por TokenProfile (US-035, TOK-TC-001..004)

Por qué no se cubre: Crear documentos TokenProfile exige is_admin de plataforma, sin vía alternativa (403 con la cuenta QA). Sin dos perfiles (con y sin media.category) no se puede ejercitar el gate por módulo del PR sm2#8451. Distinto del scope read/write, que sí es self-service.
Cómo cubrirlo: Setup manual una sola vez: un admin crea los 2 TokenProfile + ids en .env. Ver TOK-RISK-001.

gap GAP-DISTRIBUTION-TOKENS — Distribution Tokens (sección aparte de /settings/api)

Por qué no se cubre: La pantalla tiene una segunda sección distribution-token-* (con category/live/ip access) no explorada en esta sesión; foco fue el token de cuenta read/write.
Cómo cubrirlo: Explorar en una sesión futura si es prioritario para el negocio.

Hallazgos (5)

medium bug AQ2#55 — Token read-only: POST /api/show devuelve 500 en vez de 403

Un token de solo lectura que intenta escribir en /api/show recibe 500 {"message":null}, mientras que media, live-stream y playlist devuelven el 403 correcto. El gate de escritura por token en la ruta del nuevo módulo de shows lanza una excepción no capturada en vez del 403 estándar. Verificado en vivo.
Recomendación: Devolver 403 (consistente con el resto de recursos). Prueba viva @TOK-TC-011 (test.fail) en rojo-esperado hasta el fix.

low ux MEJORA-form — Crear token por UI no da feedback y no limpia el formulario

Tras Create New Token no hay toast/mensaje de éxito (solo aparece la fila) y el campo Description conserva lo tecleado → invita a crear duplicados. Nielsen #1 (visibilidad de estado) / #5 (prevención de errores). No bloqueante.
Recomendación: Mostrar confirmación de creación y resetear el form tras éxito. (TOK-RISK-004)

low ux MEJORA-desc — Se puede crear un token sin descripción

POST /api/account/token sin description responde 200 y crea el token → tokens difíciles de auditar ('¿de qué integración es este?'). No bloqueante.
Recomendación: Considerar exigir descripción/etiqueta al emitir un token (al menos en la UI). (TOK-RISK-005)

low security-smell MEJORA-access — access estructurado cae a read-only en silencio

El server solo activa write si access === 'write' (string). Un access como objeto {read,write} se ignora y el token queda read-only, con 200 sin aviso. Falla hacia el lado seguro (menos privilegio) pero silencioso: un integrador cree que hizo un token write y no. No bloqueante.
Recomendación: Rechazar con 400 un access no reconocido, o documentar el formato exacto. (TOK-RISK-006)

low ux MEJORA-msg — Rechazo por falta de scope dice INVALID_TOKEN

Un token read-only en un POST recibe 403 {"data":"INVALID_TOKEN"}. El token no es inválido — es válido y activo, solo le falta write scope. El código confunde al integrador (sospecha de token mal/expirado). El comportamiento (403) es correcto; el mensaje engaña. No bloqueante.
Recomendación: Devolver un código específico (INSUFFICIENT_SCOPE / READ_ONLY_TOKEN). (TOK-RISK-007)

Recomendaciones

  1. Corregir AQ2#55 (500 → 403 en POST /api/show con token sin write scope): rompe el contrato de autorización.
  2. Aplicar las mejoras de UX/DX no bloqueantes: feedback+reset del form, código de error específico para falta de scope, y validar el formato de access.
  3. Gestionar el setup admin (2 TokenProfile + ids en .env) para desbloquear la cobertura del gate por perfil (US-035).
