Sesión de exploración + cobertura del módulo MoAI Options (/settings/ai): configuración de IA de la cuenta (prompts, transcripción con Deepgram/Whisper, imágenes con Gemini, metadata/highlights/chapters, MCP tokens), historia US-037.

Enfoque metodológico (el foco de la sesión): cómo se le hace QA a la IA. La clave es separar dos capas y usar oráculos distintos:
- Determinista (CRUD de prompts, config, validación) → aserción exacta, como cualquier módulo.
- No-determinista (invocación viva a OpenAI/Deepgram/Gemini) → aserción de CONTRATO (estructura de respuesta + ciclo completed + modelo + duración), nunca del contenido generado (flaky por diseño y consume cuota del proveedor). Validar *calidad* del output sería un eval offline aparte, fuera de la regresión.

Veredicto: sin bugs funcionales. Contrato de prompts (list/create/delete/improve) y de MCP Tokens (list API + CRUD por UI) verificado en vivo. 8 tests verdes (6 API + 1 UI smoke + 1 UI regression), incluyendo 1 invocación IA viva de contrato (con guard QA_SKIP_AI_LIVE para no quemar cuota en CI).

Cierre de gaps: se cerró MCP Tokens (list + CRUD UI) y transcripción de audio (Deepgram/Whisper) — con un fixture aiAudioMedia que sube el audio de prueba por ingesta remota (Dropbox dl=1), espera transcoding y limpia. La transcripción se cubre por contrato del disparo async (200 TRANSCRIPTION_PROCESSING + jobId), no por el texto. Total: 10 tests verdes (2 invocaciones IA vivas: improve + transcription).

Hallazgo de flujo (imagen): una imagen no entra por el flujo de media — media/upload?type=remote la rechaza (INVALID_FILE_EXTENSION); las imágenes van por /api/image y el ai-image de Gemini genera imágenes (no analiza una subida). Por eso la cobertura de imagen requiere un flujo distinto, pendiente de explorar.

Qué se probó

  • CRUD de prompts (determinista): GET/POST/DELETE /api/settings/ai/prompts — crear, listar, borrar.
  • Validación de la invocación IA: improve sin campos requeridos → 400 (sin crash).
  • Invocación IA viva (contrato): improve → OpenAI, asertando estructura {status:completed, model, duration, prompt} sin tocar el contenido.
  • Carga de la UI (/settings/ai): MCP Tokens, AI Audio Transcription, AI Images (Gemini), guardado global.
  • Mapa del módulo: 10 secciones, superficie de endpoints y modelos de IA cosechados en vivo.

Qué NO se cubrió (huecos de cobertura) (3)

gap GAP-HIGHLIGHTS-CHAPTERS — highlights y chapters (dependen del resultado async de la transcripción)

Por qué no se cubre: highlights/chapters consumen la transcripción ya generada del media. Encadenar subir→transcodificar→transcribir (async, minutos)→highlights en un solo test sería lento y frágil. La transcripción quedó cubierta hasta el disparo (jobId), no hasta el resultado.
Cómo cubrirlo: Cubrir con un media pre-transcrito estable (sembrado una vez) o esperando el resultado async de transcription con un poll de timeout amplio, con guard QA_SKIP_AI_LIVE. Contrato: acepta el trabajo + estructura, no el contenido.

gap GAP-IMAGEN — Features de imagen (ai-image/Gemini) — flujo distinto al de media

Por qué no se cubre: Una imagen no se sube como media transcodificable (media/upload?type=remote → INVALID_FILE_EXTENSION); va por /api/image. Y ai-image (Gemini) genera imágenes, no analiza una subida. El flujo de imagen es otro y no se pudo cubrir con el patrón de media.
Cómo cubrirlo: Explorar /api/image (upload/gestión) y el flujo de generación ai-image (Gemini) por separado, capturando sus contratos reales en vivo.

gap GAP-MODELOS — Persistencia de modelos de IA por feature (metadata/highlights/article/i18n/chapters/transcription)

Por qué no se cubre: Cada feature permite configurar un modelo custom via un modal propio (add/edit). Capturar el contrato de cada uno es laborioso y de valor nicho; se difirió frente a MCP, prompts y transcription.
Cómo cubrirlo: Capturar el contrato de guardado por UI (como se hizo con prompts/improve) y cubrir la persistencia de al menos un feature representativo en una próxima iteración.

Hallazgos (1)

info methodology METODO-IA — Estrategia de QA para features de IA: contrato, no contenido

Para las invocaciones a proveedores de IA (Deepgram, Gemini, OpenAI) no se aserta el contenido generado (es no-determinista y quema cuota). Se aserta el contrato: HTTP 200, estructura de respuesta (status:'completed', model, duration, output no-vacío) y el ciclo async. La validación de *calidad* del output pertenece a un harness de evals offline (dataset fijo + métricas/LLM-judge), separado de esta suite de regresión. Los tests de invocación viva llevan guard QA_SKIP_AI_LIVE + prodGuard para controlar costo.
Recomendación: Extender esta estrategia de contrato al resto de invocaciones vivas (ver hueco de cobertura).

Recomendaciones

  1. Extender la cobertura de contrato a highlights, chapters (dependen de la transcripción async) y al flujo de imagen (ai-image/Gemini + /api/image) — cerrados improve y transcription en esta sesión.
  2. Mantener la separación determinista/no-determinista como estándar para todo QA de IA del producto.
  3. Si se requiere validar la *calidad* de la salida de IA, montar un harness de evals offline (dataset + métricas), independiente de la regresión funcional.
