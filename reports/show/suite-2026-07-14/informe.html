Corrida completa de la suite show (npx playwright test show) después de corregir la causa raíz de los 53 fallos previos.

Veredicto: suite verde — 77 passed, 0 failed, 2 rojo-esperado (bugs vivos #25/#26), 5 skipped por precondición.

Causa raíz encontrada y corregida: el CRUD de /api/show (y /api/media) se autentica con el header X-API-TOKEN de cuenta, no con la cookie de sesión del login UI. La suite mandaba la cookie -> 401 en 45 casos. El token ya existía en .env (API_TOKEN) pero nadie lo consumía: env.js no lo exponía y el fixture api solo usaba storageState.

Fix aplicado: (1) env.apiToken en env.js; (2) nuevo fixture apiToken (APIRequestContext con X-API-TOKEN); (3) specs show-API repuntados de api a apiToken; (4) el beforeAll de episodes ahora crea 2 medias dedicadas (self-contained) en vez de cazar medias libres del listado (que en cuentas con data real ya están ocupadas -> found 0).

Qué se probó

  • UI Smoke — carga del módulo Show para usuario autenticado.
  • UI Listado — búsqueda básica, limpiar búsqueda, panel de búsqueda avanzada (toggle).
  • UI Seasons — regresión de temporadas en el detalle.
  • API Show CRUD — create/read/delete + validaciones (contra endpoints no expuestos en este build).
  • API Seasons — CRUD de temporadas (/api/show/:id/season).
  • API Episodes / Recommended / Contract — estructura y casos negativos.

info infra AUTH-FIX — Los endpoints show-API exigen X-API-TOKEN, no la cookie de sesión

El CRUD de /api/show y /api/media acepta el header X-API-TOKEN de cuenta y rechaza (401) la cookie del login UI. Antes bloqueaba 45 tests. Resuelto con el fixture apiToken. La cuenta usa el nuevo módulo de shows (activate_new_show_module), lo que explica el gateo de la auth de sesión.
Recomendación: Usar el fixture apiToken para todo endpoint que exija token de cuenta; mantener api (sesión) para lo autorizado por cookie.
