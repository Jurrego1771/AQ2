Corrida completa de la suite live-stream (filtro npx playwright test live-stream: specs smoke + api/integration/contract + regression cuyo archivo lleva live-stream en el nombre). Objetivo: dejar constancia de qué se probó, no diagnosticar.

Veredicto: 0 fallos reales de producto. 85 passed, 8 rojo-esperado (bugs conocidos con prueba viva), 3 skipped por precondición. La corrida bruta reportó 8 "failed", pero al reejecutarlos en aislamiento los 8 resultaron verdes o rojo-esperado: 6 eran *flaky* de entorno (verde aislado) y 2 (TC-11 #18, TC-13 #23) son test.fail vivos mal clasificados bajo carga. La suite corre fullyParallel contra dev compartido, así que rotan casos flaky entre corridas — no son bugs de producto. Los estados de esta tabla están reconciliados con reejecución aislada (ver notas por caso).

Qué se probó

  • Smoke — carga del módulo Live para usuario autenticado (LIVE-TC-1).
  • Listado / filtros — filtro por tipo Video, búsqueda con y sin coincidencias, restauración (LIVE-TC-2..4).
  • API CRUD — create/read/update/delete de live-stream y naming (LIVE-TC-5..7, 25).
  • Schedules — contrato API de schedule-job, audio/song metadata, edge cases, validación y auth (LIVE-TC-8..17, 62..76, 124..127).
  • Detalle — endpoints avanzados — search, toggle-online/bookmark, thumbs, logo, dvr, assign ad, players (LIVE-TC-32..53).
  • Detalle — secciones inexploradas — cosecha de marcas sm:, EPG/PlayAnywhere, distribution policy (LIVE-TC-116, 118..123).
  • Logo / Thumbnail — regresión de subida y posición (LIVE-TC-54..61).
  • Quizzes — CRUD + send + auth de quizzes del live (LIVE-TC-77..91).
  • Ingest / RTMP — contrato de ingest y push RTMP con ffmpeg (LIVE-TC-107, 111..114).

Qué NO se cubrió (huecos de cobertura) (3)

gap GAP-DEFERRED — 3 casos diferidos y eliminados de la suite el 2026-07-14 (LIVE-TC-26, LIVE-TC-40, LIVE-TC-117)

Por qué no se cubre: No eran bugs de producto sino fragilidad de la propia suite bajo fullyParallel: TC-26 (F4 ResourceSweeper) barría por runId compartido y borraba recursos de workers hermanos → timeout 30s; TC-40 (pagination) exigía IDs idénticos entre dos list() separados → un create concurrente desplaza la ventana; TC-117 (Start Recording disabled) hacía timeout bajo carga por estado transitorio del botón durante la carga. Los tres pasan en aislamiento (7.5s / diff confirmado / 9.1s).
Cómo cubrirlo: Rehacerlos bien: TC-26 con marcador único por test (no el runId compartido); TC-40 con snapshot único sin 2ª llamada; TC-117 esperando estado estable del detalle. Definiciones conservadas en knowledge-core con estado: deferred.

gap GAP-FLAKY-ENV — Inestabilidad de ejecución en dev compartido bajo fullyParallel

Por qué no se cubre: Entre corridas, 4-8 casos fallan por latencia/concurrencia (timeouts de page.goto/XHR, desfases de conteo de listado, 5xx transitorios) y pasan en el reintento local o en aislamiento. El set de casos afectados rota — no es determinístico ni atribuible a un caso concreto.
Cómo cubrirlo: Reducir workers para regresión UI, o aislar en su propio worker los tests que mutan datasets compartidos (listado/paginación). El Semaphore (permits=2) ya mitiga los 502; evaluar bajarlo aún más para la capa regression.

gap GAP-SKIPS — Skips por precondición de entorno/datos (no son fallos)

Por qué no se cubre: Casos que se saltan por falta de precondición: ffmpeg no en PATH (signal), no hay ads/players en la cuenta (assign-ad / player), sin API para materializar EventSchedule self-contained (schedule-edge), y prodGuard (env.isProd) para escrituras.
Cómo cubrirlo: Instalar ffmpeg en el runner; sembrar ads/players de prueba en la cuenta dev; pedir a dev un endpoint/fixture para EventSchedule. Los prodGuard son correctos y se dejan.
