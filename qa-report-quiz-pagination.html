Reporte original

El historial de quizzes enviados en el detalle de Live no tenía scroll ni paginación — con muchas entradas (100+) se volvía inutilizable. Verificado además por lectura de código: el backend viejo ya limitaba a 100 registros sin avisar, así que en cuentas con más de 100 quizzes históricos los más antiguos eran invisibles en silencio incluso antes de este bug de UI.

Método

Se creó un live-stream y un media descartables por API (borrados al finalizar):

  • Confirmación de despliegue: se buscaron las marcas sm: nuevas del PR en el DOM.
  • 15 quizzes creados en el live-stream (vía UI para capturar el payload real, luego por API directa) para ejercitar 3 páginas completas.
  • Batería de 10 llamadas directas a GET .../quizzes cubriendo el contrato completo y sus casos límite.
  • Interacción real con los controles de paginación (botones, selector de tamaño, borrado en cascada) en el browser.
  • 3 quizzes en el media, con timestamps espaciados exactamente para probar el límite de la validación de intervalo (server-side, antes era client-side).

Hallazgo menor (no bloqueante)

Inconsistencia de nombre de campo id entre creación y listado

POST .../quizzes devuelve el quiz creado con campo id; GET .../quizzes devuelve _id (documento Mongoose crudo, sin transformar). No es un bug para el usuario final (la UI ya maneja esto correctamente), pero sí una inconsistencia de contrato a tener en cuenta al escribir clientes/tests contra esta API — mi primer intento de editar un quiz por API falló con un error 500 de Mongoose por usar quiz.id (undefined) en vez de quiz._id del listado.

Sin verificar / fuera de alcance de esta pasada

El loader visual (quiz-loader) no se capturó en pleno estado de carga (requiere retrasar artificialmente la respuesta, no se intentó). El flujo de "enviar" un quiz durante un live (ícono de avión de papel, bloquea edición/borrado tras enviarse) no se probó — fuera del alcance de este PR específico de paginación.
