La alerta "editado por otro usuario" funciona correctamente en Live y Media

Entorno: dev (dev.platform.mediastre.am) Origen: mediastream/sm2#8317 Fecha: 2026-07-03
✓
Verificado con dos sesiones reales — funciona como se espera 4 aserciones confirmadas en vivo: alerta cruzada (2 módulos) + no-autoalerta (1 módulo, 2 guardados).

Qué se pidió verificar

Sistema implementado para avisarle a un usuario, mediante un banner, que otro usuario editó el mismo registro mientras tenía la página abierta — recomendando refrescar. Aplica a live streams, medias y artículos. Se pidió comprobarlo con dos sesiones reales en paralelo (botqa@mediastre.am como el usuario que ve la alerta, botcito2@mediastre.am como el que edita).

Método

  1. 1. Dos pestañas del mismo browser comparten cookies (mismo cookie jar) — no alcanza con abrir una segunda pestaña para tener dos sesiones. Se logueó/desлогueó explícitamente en cada pestaña para controlar qué usuario tenía la sesión activa en cada momento.
  2. 2. Orden crítico: la pestaña "espectadora" debe cargar la página primero, con la cookie del usuario que debe ver la alerta — el userId de comparación queda embebido en el JS de esa carga (no se re-consulta después). Recién después se cambia la sesión de la otra pestaña al segundo usuario y se guarda.
  3. 3. Se creó un live-stream y un media descartables por API (autocontenidos, borrados al final) — nunca se usaron recursos existentes.
  4. 4. Se interceptó utils.send_alert en tiempo de ejecución (parche sobre el módulo compartido, vía window.__modules) para capturar cada invocación de forma determinística, además de confirmar visualmente con captura de pantalla.

Resultado por aserción

Artículos no se probó en vivo por alcance/tiempo — mismo patrón de código exacto (article/detail.coffee + article/update.js, revisado en la sesión de análisis del PR), alto grado de confianza por analogía directa.

Evidencia técnica (interceptor de send_alert)

Además de la captura visual, se interceptó la función real para registrar cada llamada — evita falsos negativos por selectores CSS incorrectos (el banner usa position: fixed, que hace que offsetParent reporte null aunque esté visible; se confirmó con getBoundingClientRect en su lugar).

Nota metodológica (para quien repita esto)

El orden de sesión/carga importa

En un primer intento, se navegó a la página de media después de que la otra pestaña ya hubiera logueado al segundo usuario — como las cookies son compartidas por todo el browser (no por pestaña), esa carga fresca quedó embebida con el userId equivocado. Se corrigió re-logueando al usuario espectador y recargando antes de tocar la sesión del editor. Documentado en knowledge-core/cross-cutting/ si se decide dejar precedente para futuras pruebas de este tipo con dos sesiones.
